# 项目流程选择

本文件提供用户可见的流程选项。先按 `references/flow-clarification-protocol.md` 给出基于目标、材料和风险的建议；不得把建议当作用户确认。用户可用数字选择、改选或自定义节点。

## 选择项

| 数字 | 用户选项 | 结果 |
| --- | --- | --- |
| `1` | 小型需求/BugFix | 采用最小的需求或缺陷修复流程。 |
| `2` | 中型项目 | 增加影响分析、架构和双环境配置规划。 |
| `3` | 大型系统 | 采用完整规格、独立验证、Evidence、LOOP 与审计链。 |
| `4` | 自定义流程 | 用户明确保留、删除或调整节点后，再确认流程。 |
| `5` | 直接改 | 不创建正式 Plan 或 Teams；只修改指定内容并运行适用轻量验证。 |

选项 `1`、`2`、`3` 分别规范化为 `small`、`medium`、`large`，确认后才允许创建正式 Plan、Teams 规划表或派发成员。选项 `4` 先进入自定义节点澄清；选项 `5` 对应最小请求路径，不得借此绕过安全、授权、发布或生产边界。

## 流程与编制参考

### 小型需求/BugFix

适用于单点、低风险且不涉及跨模块、生产发布或安全边界的需求或 BugFix。

`需求/原型（如有 UI）或 BugFix 定位 → 测试用例与造数 → TDD 与实现 → 测试及独立复核`

- 项目成员：通常 1–2 人。
- 可能的 Subagent：0–1 个。
- 常见角色：Goal Lead、实现/测试与轻量独立复核。

### 中型项目

适用于多模块、API 或数据边界、原创 UI，或开发与生产环境存在差异的项目。

`需求卡、PRD 与影响分析 → 架构设计及开发/生产配置规划 → 原型、测试与造数 → TDD、实现与集成 → 独立测试及 Review`

- 项目成员：通常 3–5 人。
- 可能的 Subagent：2–4 个。
- 常见角色：需求/产品、架构或开发、测试、QA/Reviewer。

### 大型系统

适用于多系统协作、发布、生产变更、安全敏感、支付/认证或复杂 UI 等高风险系统。

`流程澄清、需求规格与 PRD → 页面原型及架构 → 开发/生产配置规划 → 独立测试设计、Harness 与实现 → Evidence、Review、LOOP 补缺及完成审计`

- 项目成员：通常 6–10 人。
- 可能的 Subagent：5–8 个。
- 常见角色：Goal Lead、需求/产品、架构、前后端、测试设计与执行、QA、Reviewer、Auditor，以及按需接入安全或性能专家。

这些是规划参考范围，必须按任务范围、风险和可用运行时能力调整。涉及安全、发布或生产边界时，不得因选择小型流程降低必要门禁。
