# 需求卡片：登录页空状态提示 V0.1

## 核心目标

- 为演示项目规划一个登录页空状态提示，不修改业务代码，只生成文档、HTML 原型和最小 Harness 复盘资料。

## 关键功能

- 明确登录页空状态的文案、触发场景和用户下一步动作。
- 生成可复盘的 SPEC、HTML 原型、测试计划和验收清单。
- 记录 `setup -> run -> checks -> report` 的最小 Harness 链路。

## 用户故事

| ID | 用户故事 | 价值 |
| --- | --- | --- |
| US-001 | 作为未登录用户，我想要看到清楚的空状态提示，以便知道当前没有可恢复会话并继续登录。 | 减少空白页面带来的困惑 |
| US-002 | 作为新用户，我想要看到创建账号或联系支持入口，以便在没有账号时知道下一步。 | 降低新用户转化阻力 |

## 功能验收标准

| ID | 对应功能/故事 | 验收标准 |
| --- | --- | --- |
| AC-001 | US-001 | Given 登录页没有可恢复会话，When 页面展示空状态，Then 用户能看到空状态标题和辅助说明。 |
| AC-002 | US-001 | Given 空状态已展示，When 用户查看主操作，Then 主按钮文案为“登录”。 |
| AC-003 | US-002 | Given 用户没有账号，When 查看次操作区域，Then 能看到“创建账号”或“联系支持”入口。 |
| AC-004 | 示例边界 | Given 本示例只生成文档和原型，When 验收原型，Then 不出现真实认证调用。 |

## 边界

- 范围内：版本目录文档、SPEC、HTML 原型、Harness 静态样例。
- 范围外：真实登录接口、认证逻辑、生产部署、业务代码实现。
- 禁止/只读：不得修改示例项目的业务实现文件。

## 约束

| 约束 | 影响 | 处理方式 |
| --- | --- | --- |
| 示例只做文档和原型 | 无法提供真实运行截图或后端测试 | Harness 标记为静态样例，并记录 `sample_only` |
| 缺少业务项目指南 | 风格与验收只能基于默认规则 | 使用 `references/default-AGENTS.md` 作为默认指南 |
| UI 不进入真实浏览器运行 | E2E 证据不能标记为真实通过 | 在 acceptance 中说明未执行真实 E2E |

## 风险

| 风险 | 影响 | 缓解方式 | 是否需确认 |
| --- | --- | --- | --- |
| 把示例误认为真实交付 | 误判 Harness 和验收强度 | 文件中标明 `sample_only` 和 `no_runner` | 否 |
| 空状态业务语境不足 | 文案可能不贴合真实业务 | 在需求规格卡保留待确认问题 | 是 |
| HTML 原型无法验证真实状态 | 只能表达结构和文案 | 后续真实项目中补 E2E 和截图证据 | 否 |

## 待确认问题

1. 空状态是否面向未登录、登录失败、账号未开通，还是登录页无可用组织？
2. 是否需要品牌语气、合规提示或帮助入口？
